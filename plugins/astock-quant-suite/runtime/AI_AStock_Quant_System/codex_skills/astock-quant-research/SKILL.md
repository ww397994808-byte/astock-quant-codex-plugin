# A股量化研究

Use this skill to move an A股 strategy idea through a gated research workflow. Treat every step as evidence collection, not as permission to trade.

## Project Root

Default to the current workspace when it contains `cli.py`. If the skill was installed with `scripts/install_astock_skill.py`, the helper script also reads `project_root.txt`, which records the project path used during installation. Before running commands, confirm the directory contains `cli.py`.

## Non-Negotiable Gates

1. Start with intake. Do not backtest a raw idea before generating `backtest_plan.yaml`.
2. Enforce signal causality. A signal at bar `t` may only use current and earlier K-line data. Any future data, centered rolling window, negative shift, future label, same-bar close fill, or post-event adjustment leak is `INVALID`.
3. Match the backtest to the strategy archetype. Single-symbol timing, grid, rotation, stock selection, weekly swing, and intraday strategies do not share the same assumptions.
4. Use `stage-check` after every major run. If the stage is blocked, explain the blocker and stop promotion.
5. Treat QMT as readonly first. `qmt-check` only proves account, cash, positions, orders, and trades can be read. It never means real orders are allowed.
6. Require explicit manual confirmation before any real-trade path. Do not help bypass `pretrade-check`.
7. Keep asset versions separate. This skill is the A股/QMT version only. If the idea mentions BTC, ETH, USDT, crypto exchanges, perpetuals, or digital currency, stop and explain that the crypto version needs a separate workflow; do not reuse A股 T+1, price-limit, adjustment, or QMT assumptions.

## Standard Workflow

Run commands from the project root:

```bash
python3 cli.py student-start --session-id <学员或案例ID>
python3 cli.py student-doctor
python3 cli.py student-course-path --idea "<策略想法>" --session-id <学员或案例ID>
python3 cli.py student-research-contract --idea "<策略想法>" --session-id <学员或案例ID>
python3 cli.py student-contract-check --contract <契约报告目录> --workflow <workflow报告目录>
python3 cli.py student-first-run --idea "<策略想法>" --session-id <学员或案例ID>
python3 cli.py student-handoff-pack --session-id <学员或案例ID>
python3 cli.py student-idea-preflight --idea "<策略想法>" --session-id <学员或案例ID>
python3 cli.py student-backtest-plan-precheck --idea "<策略想法>" --session-id <学员或案例ID>
python3 cli.py student-future-leak-precheck --file <策略文件.py>
python3 cli.py core5-walk-forward --out reports/core5_relative_strength_grid_package --starts 2018,2019,2020,2021
python3 cli.py student-product-audit
```

Use `student-start` as the one-command beginner entrypoint when the learner asks "where am I now" or "can I continue". It runs student doctor, control center, optional session index, and a dry-run next-step preview when the next command is safe. It writes `STUDENT_START.md/json` and `student_start_cards.json`. It must not execute the next research command, connect QMT, or place orders.
Use `student-doctor` at the start of a beginner session or after installing/updating the project. It checks whether the project, beginner commands, skill package, sample data, and QMT safety config are ready without connecting QMT, running backtests, or placing orders. If it returns `BLOCKED_ENVIRONMENT`, fix those blockers before research.
Use `student-course-path` as the preferred zero-to-research route for course delivery. It runs student doctor, idea preflight, backtest plan precheck, and optional future-leak code precheck when `--file` or `--code` is supplied, then writes `STUDENT_COURSE_PATH.md/json` and `student_course_path_cards.json` with the route steps, blockers, warnings, source reports, and next command. It must not execute `student-workflow`, fetch data, connect QMT, pretrade, handoff, sandbox, or place orders.
Use `student-research-contract` after the route is clear and before running `student-workflow` when the learner or mentor needs a fixed pre-run contract. It runs the course route, then writes `STUDENT_RESEARCH_CONTRACT.md/json`, `student_research_contract_cards.json`, and `research_contract.json` with a stable `contract_id`, asset boundary, symbol, strategy pattern, timeframe, adjustment, execution model, audit requirements, future-leak rule, and required gates. It must block if the course path is blocked or code future-leak precheck fails. It must not execute research, connect QMT, pretrade, handoff, sandbox, or place orders.
Use `student-contract-check` after `student-workflow` when the learner or mentor needs to verify the workflow still matches the research contract. It accepts `--contract <契约报告目录或JSON>`, `--workflow <workflow报告目录或workflow_manifest.json>`, or just `--session-id <id>` to use latest matching evidence. It writes `STUDENT_CONTRACT_CHECK.md/json` and `student_contract_check_cards.json`, and blocks if symbol, asset type, strategy pattern, timeframe, adjustment, template, or execution model drifted. It must not run research, connect QMT, pretrade, handoff, sandbox, or place orders.
Use `student-first-run` for a beginner's first session when they have an idea and want the safest one-command path. It runs student doctor and idea preflight, then prepares the exact `student-workflow` command. By default it does not execute the workflow; only `--execute` runs the safe research workflow. It must never connect QMT, pretrade, handoff, sandbox, or place orders.
Use `student-handoff-pack` when the mentor or assistant needs to hand a learner one consolidated package after a session. It runs read-only start/status, optional session report, and optional product audit, then writes `STUDENT_HANDOFF_PACK.md/json` and `student_handoff_cards.json` with current stage, next command, report links, learner checklist, and hard boundaries. It must not execute research, connect QMT, pretrade, handoff, sandbox, or place orders.
Use `student-idea-preflight` before a beginner's first `student-workflow` when the idea may be vague, when the mentor wants a one-page quality gate, or when asset/version boundaries need to be checked. It writes `STUDENT_IDEA_PREFLIGHT.md/json` and `student_idea_cards.json`, resolves the A-share symbol, checks completeness, builds the backtest plan, rejects crypto/digital-currency ideas for this A-share version, and returns either a safe `student-workflow` command or the next `intake-chat` clarification command. It must not run the workflow, fetch data, connect QMT, or place orders.
Use `student-backtest-plan-precheck` when the learner or mentor needs to verify whether a strategy's archetype, timeframe, backtest template, data requirements, execution assumptions, audit requirements, and QMT promotion boundary match before running a full workflow. It accepts `--idea`, optional `--timeframe`, `--strategy-pattern`, `--adjust`, `--session-id`, and `--case-id`, then writes `STUDENT_BACKTEST_PLAN_PRECHECK.md/json`, `student_backtest_plan_cards.json`, and `backtest_plan_precheck.yaml`. It must block unsupported archetypes, bad timeframe/archetype combinations, event-driven plans without point-in-time event availability, and non-A-share asset versions. It must not run a backtest, fetch data, connect QMT, or place orders.
Use `student-future-leak-precheck` when the learner pasted strategy code, edited a strategy file, or asks whether the code has future functions before running a full workflow. It accepts `--file <策略文件.py>` or `--code "<代码>"`, writes `STUDENT_FUTURE_LEAK_PRECHECK.md/json`, `student_future_leak_cards.json`, and `submitted_strategy.py`, and blocks obvious future data, centered windows, negative shifts, future labels, dangerous imports/IO, or non-causal signal code. It is a static precheck only; a `LEAK_CHECK_VALID` result still requires the normal workflow audit, paper observation, and stage-check.
Use `core5-walk-forward` only for the bundled Core5 relative-strength grid package. It starts the strict walk-forward backtest directly, writes `start_<year>_decisions.csv`, `start_<year>_walk_forward_audit.json`, and `report.md`, and must fail if `param_train_end_date > rebalance_date` or `test_start_date <= rebalance_date`. Treat its output as fixed-pool research evidence only, not as a general student idea workflow or trading permission.
Use `student-product-audit` when the mentor asks whether the project is ready for course delivery, what is still missing, or whether a local machine can be handed to learners. It reads environment checks, command registration, docs, skill files, tests, QMT safety config, latest workflow reports, and session evidence, then writes `STUDENT_PRODUCT_AUDIT.md/json` and `student_product_cards.json`. It is read-only and must not be treated as research evidence or trading permission.
If `student-doctor` says `config/qmt_config.yaml` is missing, use `qmt-config-init` to create a local safe config. This command must force `dry_run=true` and `enable_real_trade=false`; it must not connect QMT, place orders, or enable real trading. Use optional `--account-id` and `--mini-qmt-path` only to prepare readonly checks.

```bash
python3 cli.py qmt-config-init
python3 cli.py qmt-config-init --account-id "<QMT账号>" --mini-qmt-path "<miniQMT路径>" --force
python3 cli.py qmt-config-status
```

```bash
python3 cli.py student-workflow --idea "<包含标的的策略想法>" --timeframe <周期> --adjust point_in_time_qfq --session-id <学员或案例ID>
```

This command avoids manual run-id copying and writes `workflow_manifest.json`, `STUDENT_WORKFLOW_SUMMARY.md`, `NEXT_ACTIONS.md`, `STUDENT_ACCEPTANCE_CHECKLIST.md`, `STUDENT_DIAGNOSTICS.md`, `BACKTEST_ASSUMPTION_CARD.md/json`, and, when a paper policy card exists, `STUDENT_POLICY_ACTION_PLAN.md/json`.
Use one stable `--session-id` per learner or case when teaching. The session id is written into the workflow manifest, repair DSL, safe runner ledger, and session report. Reuse the same session id with `student-control-center`, `student-run-next`, `student-safe-loop`, and `student-session-report` to avoid mixing reports from tests, demos, or other learners. Use `--case-id` when the same learner has multiple strategies.
Use `student-session-index` when the mentor needs to see all known learners/cases on the current machine. It reads workflow manifests, backtest assumption cards, paper observation policy cards, and the safe runner ledger, then writes `STUDENT_SESSION_INDEX.md/json` and `student_session_cards.json` with each session's latest workflow, current stage, backtest assumptions, paper policy status/failed metrics, paper policy repair hints, ledger counts, risk notes, and next control-center command. It is read-only and must not be treated as research evidence or trading permission.
If `--strategy` is omitted, the workflow selects a registered default strategy from the intake plan and idea text.
If `--symbol` is omitted, the workflow resolves the symbol from the idea text, such as `中国神华 -> 601088.SH`.
If `--data` is omitted, the workflow fetches or prepares deterministic local sample-backed data.
Use `--auto-refine` when teaching beginners. It runs one guarded second attempt by taking the first run's blockers and appending missing exit, sizing, risk, or paper-observation constraints to the idea.

The bundled helper script delegates to the same beginner workflow:

```bash
python3 ~/.codex/skills/astock-quant-research/scripts/run_astock_workflow.py --idea "<策略想法>" --timeframe 1d --adjust point_in_time_qfq --auto-refine --session-id <学员或案例ID>
```

```bash
python3 cli.py intake --idea "<策略想法>"
python3 cli.py fetch-data --symbol <代码> --timeframe <周期> --adjust point_in_time_qfq

python3 cli.py qmt-config-status
python3 cli.py qmt-check
python3 cli.py stage-check --run-id <paper_run_id> --plan-run-id <intake_run_id> --qmt-run-id <qmt_run_id>
python3 cli.py pretrade-check --strategy <策略名> --symbol <代码> --run-id <paper_run_id> --plan-run-id <intake_run_id> --qmt-run-id <qmt_run_id>

```

This creates `compile_report.md`, `compiled_strategy.json`, `audit_report.md`, `future_leak_report.md`, `readiness_report.md`, `paper_observation_report.md`, `paper_observation_policy_card.json`, `stage_gate_report.md`, `repair_dsl_run_report.md`, `repair_dsl_next_actions.md`, and, when needed, `repair_dsl_auto_repair.md/json`. When an auto-repair candidate is ready for QMT readonly, it also creates `REPAIR_DSL_PROMOTION.md/json` and `promotion_candidate/`.

## Reference Routing

- Read `references/future_leak_rules.md` when reviewing strategy code, indicators, labels, execution timing, adjustment, or audit failures.
- Read `references/workflow.md` when planning a beginner-facing session or deciding the next command.
- Read `references/strategy_archetypes.md` when mapping an idea to data cycle, template, universe, or backtest assumptions.
- Read `references/paper_observation.md` when interpreting paper results or deciding whether the paper run has enough observations for its strategy type.
- Read `references/qmt_gate.md` when the user mentions QMT,实盘,下单,账户,资金,持仓, or broker connection.

## Response Style

Give the learner the current stage, the blocking reason, and the next executable command. Avoid big conceptual lectures unless the user asks; the value of this skill is reducing ambiguity at each gate.
When the learner asks whether the project is installed correctly, whether they can start from zero, or why the first run cannot begin, run `student-doctor` first. Read `STUDENT_DOCTOR.md/json` and use its `status`, `summary`, blockers, warnings, and `next_commands` as the answer. Treat unsafe QMT config such as `dry_run=false` or `enable_real_trade=true` as a stop state, and do not change those switches automatically.
When a zero-basis learner or course participant asks how to start from one idea, prefer `student-course-path --idea "<策略想法>" --session-id <id>` before `student-first-run`. Read `STUDENT_COURSE_PATH.md/json` and `student_course_path_cards.json`; use `status`, `route_steps`, `blockers`, `warnings`, `sources`, `safe_to_copy`, and `next_command` as the answer. If `--file` or `--code` is available, include it so future-leak code precheck is part of the route. If it returns `COURSE_PATH_READY`, give the generated `student-workflow` command but do not execute it unless explicitly asked. If it returns `COURSE_PATH_BLOCKED`, explain the first blocker and report path. If it returns `COURSE_PATH_NEEDS_CLARIFICATION`, ask for the missing information instead of starting research.
When the learner is about to start a formal research branch, or the mentor wants to prevent assumption drift, run `student-research-contract --idea "<策略想法>" --session-id <id>` before `student-workflow`; include `--file` or `--code` if strategy code exists. Read `STUDENT_RESEARCH_CONTRACT.md/json`; use `status`, `contract_id`, `contract`, `blockers`, `warnings`, `sources`, and `next_command` as the answer. Treat `CONTRACT_READY` as permission to start research only, not as audit or trading approval. Treat `CONTRACT_BLOCKED` as a stop state. If code exists but was not attached, call out the `future_leak_not_attached` warning.
After a formal research branch has run, use `student-contract-check --contract <contract_dir> --workflow <workflow_dir>` or `student-contract-check --session-id <id>` before interpreting the workflow as evidence for that contract. Read `STUDENT_CONTRACT_CHECK.md/json`; use `status`, `checks`, `blockers`, `warnings`, and source paths. Treat `CONTRACT_MATCHED` as "the workflow obeyed the pre-run assumptions", not as performance, audit, paper, QMT, or trading approval. Treat `CONTRACT_DRIFT` and `CONTRACT_CHECK_BLOCKED` as stop states.
When the learner asks to start from one idea with minimum steps, run `student-first-run --idea "<策略想法>" --session-id <id>` first. Read `STUDENT_FIRST_RUN.md/json` and `student_first_run_cards.json`; use `doctor_status`, `preflight_status`, `allowed`, `execute`, `command`, `child_report_path`, and `reason` as the answer. If it is `READY_TO_EXECUTE`, give the command or explicitly ask before using `--execute`. If it is `BLOCKED`, use the doctor/preflight report path to explain whether the blocker is environment, idea completeness, asset version, or backtest plan. Never add `--execute` when the learner has not asked to actually run the research workflow.
When the learner or mentor asks for a shareable summary, handoff package, lesson wrap-up, or "what should I send the student", run `student-handoff-pack --session-id <id>` when possible. Read `STUDENT_HANDOFF_PACK.md/json` and `student_handoff_cards.json`; use `status`, `next_action`, `next_command`, `report_links`, `learner_checklist`, and `warnings` as the answer. Treat it as navigation only, not research evidence or trading permission.
When the learner gives a raw strategy idea and asks whether it can start, whether it is clear enough, or how to turn it into a run, use `student-idea-preflight --idea "<策略想法>"` before `student-workflow`. Read `STUDENT_IDEA_PREFLIGHT.md/json` and `student_idea_cards.json`; use `status`, `completeness_score`, `parsed`, `backtest_plan`, `blockers`, `warnings`, `clarifying_questions`, and `next_command` as the answer. If it returns `READY_FOR_STUDENT_WORKFLOW`, give the generated `student-workflow` command. If it returns `NEEDS_CLARIFICATION`, ask the listed clarification questions or run the generated `intake-chat` command. If it returns `WRONG_ASSET_VERSION`, stop and explain that the crypto version needs a separate workflow.
When the learner asks whether the backtest logic, data cycle, timeframe, or strategy archetype is appropriate, run `student-backtest-plan-precheck --idea "<策略想法>"` before any backtest. Read `STUDENT_BACKTEST_PLAN_PRECHECK.md/json` and `student_backtest_plan_cards.json`; use `status`, `backtest_plan.strategy_pattern`, `timeframe`, `template_name`, `execution_model`, `assumption_checks`, blockers, warnings, and `next_command` as the answer. If it returns `BLOCKED_BACKTEST_PLAN`, `WRONG_ASSET_VERSION`, or `MISSING_IDEA`, stop and explain the mismatch. If it returns `BACKTEST_PLAN_READY`, say that the plan can enter `student-workflow` but still needs future-leak audit, paper observation, and stage-check.
When the learner provides strategy code or asks whether code has a future function, run `student-future-leak-precheck --file <path>` or `student-future-leak-precheck --code "<code>"` before any backtest. Read `STUDENT_FUTURE_LEAK_PRECHECK.md/json` and `student_future_leak_cards.json`; use `status`, `checker_report.high_count`, findings, blockers, and `next_command` as the answer. If it returns `LEAK_RISK_FOUND` or `MISSING_CODE`, stop and explain the exact repair. If it returns `LEAK_CHECK_VALID`, say only that no HIGH static leak was found, then continue through `student-idea-preflight` or the full workflow; never treat this as final audit or trading permission.
When the mentor asks whether the project is product-ready, course-ready, suitable for learners, or what is still missing from the local delivery package, run `student-product-audit` first. Read `STUDENT_PRODUCT_AUDIT.md/json` and `student_product_cards.json`; use `status`, `can_deliver_to_students`, `blockers`, `warnings`, category cards, and `next_commands` as the answer. Treat `BLOCKED_PRODUCT_DELIVERY` as a stop state. Treat `PRODUCT_READY_WITH_WARNINGS` as usable for course/internal delivery but not finished; name the warning categories before recommending the next step.
When QMT local config is missing, suggest `qmt-config-init` before `qmt-check`. If the config already exists, do not overwrite it unless the learner explicitly asks and the command includes `--force`. Never suggest changing `dry_run` to false or `enable_real_trade` to true.
Before recommending `qmt-check`, run `qmt-config-status` unless a fresh `QMT_CONFIG_STATUS.md/json` already proves `READY_FOR_QMT_READONLY`. Read `QMT_CONFIG_STATUS.md/json` and `qmt_config_action_cards.json`; if it reports missing `account_id`, missing `mini_qmt_path`, nonexistent path, `dry_run=false`, or `enable_real_trade=true`, stop and give the next command from the report. `qmt-config-status` is read-only and is not evidence that QMT itself is connected.
When the mentor asks which learners/cases exist, which sessions are active, or how to find a learner's latest run, run `student-session-index` first. Read `STUDENT_SESSION_INDEX.md/json` and `student_session_cards.json`; use each session's `latest_workflow`, `latest_workflow.backtest_assumption`, `latest_workflow.paper_policy`, `current_stage`, `ledger_summary`, `risk_notes`, and `next_command` as the answer. Call out the backtest assumption status/pattern/timeframe first, then `paper_policy.failed_metrics` and `paper_policy.repair_hints` such as `trade_count`, `completed_rounds`, `max_drawdown`, or `rejected_order_rate` before discussing QMT. Then use `student-control-center --session-id <id>` for the selected session.
When the learner asks broadly "what should I do next", "where am I now", "can this project be used", or "what is missing" without naming a specific report, run `student-start` first. If a session id is known, pass `--session-id <id>` so reports from tests, demos, or other learners do not pollute the answer. Read `STUDENT_START.md/json` and `student_start_cards.json`; use its `status`, `current_stage`, `next_action`, `next_command`, `safe_to_copy`, `preview_status`, and any `backtest_assumption` card as the top-level answer. Use `student-control-center` directly only when the learner names a specific workflow, promotion, or QMT dashboard directory, passing it with `--workflow`, `--promotion`, or `--qmt-dashboard`. When `--workflow` is explicit and no session id exists, do not mix unrelated latest promotion or QMT dashboard evidence; use only the named workflow, its backtest assumption card, its paper observation policy card, and explicitly named evidence. If `safe_to_copy` is false, explain which placeholder or manual decision is missing.
If `student-control-center` returns `safe_to_copy: true` and the learner wants the system to continue automatically, use `student-run-next --dry-run` first, carrying the same `--session-id` when available. Only run it without `--dry-run` when the command remains in the safe whitelist. `student-run-next` must refuse QMT handoff, sandbox, pretrade, and live-trade related commands; it may only execute safe research repair, QMT readonly, stage-check, dashboard, or control-center commands. Read `STUDENT_NEXT_STEP_RUN.md/json`, `execution_decision.json`, and `reports/student_session_ledger.jsonl` after using it so the learner can see why the step was allowed or blocked.
When the mentor asks what a learner has done, whether the learner tried unsafe steps, or how to review a session, run `student-session-report --limit 20 --session-id <id>` when a session id is known. Read `STUDENT_SESSION_REPORT.md/json`; use its allowed/blocked/dry-run/executed counts and `risk_notes` as the review summary.
When the learner or mentor asks to keep going automatically across multiple safe steps, run `student-safe-loop` without `--execute` first, carrying the same `--session-id` when available. Read `STUDENT_SAFE_LOOP.md/json`; only consider `--execute` when every planned step is allowed by `student-run-next` and remains within the safe whitelist. The loop previews each step before execution and must stop at any blocked command, placeholder, repeated next command, QMT handoff, sandbox, pretrade, or live-trade related command.
When a workflow is `INVALID`, tell the learner to open `NEXT_ACTIONS.md` first. If `STUDENT_POLICY_ACTION_PLAN.md/json` exists, use it to translate paper policy failures into the next research branch command; it is not QMT or live-trading permission. `student-control-center` may choose this plan as `POLICY_ACTION_PLAN_READY`, and `student-run-next --dry-run` may preview its `student-workflow` command because it is still a safe research command.
When `BACKTEST_ASSUMPTION_CARD.md/json` exists, use it to explain why this strategy's backtest differs from another strategy's backtest. Call out `strategy_pattern`, `timeframe`, `data_required`, `execution_model`, `learner_checks`, and `promotion_policy` before suggesting QMT. Intraday, grid, rotation, stock selection, and portfolio strategies must not be treated as if they shared the daily single-symbol swing assumptions.
When the learner asks why a run is blocked, use `STUDENT_DIAGNOSTICS.md` to distinguish missing data, too few signals, excessive drawdown, rejected orders, and QMT-readonly gaps.
When `STUDENT_EXPERIMENTS.md` exists, explain that the candidates are research directions only; the learner must rerun paper observation and stage gates before any QMT step.
Use the ranked `next_command` in `STUDENT_EXPERIMENTS.md` when the learner asks what to try next. It may include `--strategy-params` JSON to reproduce the candidate exactly.
Candidate experiments may include `strategy_switch` rows such as switching from Bollinger mean reversion to MA cross or grid. Treat them as new research branches, not as proof that the old strategy is fixed.
If a ranked candidate has negative return, use its `loss_analysis` before suggesting another run. Explain whether the loss came from all losing completed rounds, fee drag, drawdown pressure, or unclear trade quality.
When `repair_actions` exist, use the generated `next_research_idea` and `next_command` as the preferred next step, while reminding the learner that these actions create a new research branch rather than a live-trading candidate.
Check each repair action's `implementation_status`. `component_available_requires_compiled_strategy` means the component exists but was not executed in the current direct strategy run; `research_note_only` means the idea text changed but code still needs implementation before evidence counts.
When `STUDENT_REPAIR_DSL.md` exists, explain it as a structured repair research branch. Prefer its `repair-dsl-backtest` command as the next executable step. It is not evidence until that DSL branch is separately backtested, audited, paper-observed, and stage-checked.
After `repair-dsl-backtest`, read `repair_dsl_run_report.md`, `repair_dsl_next_actions.md`, `audit_report.md`, `future_leak_report.md`, `readiness_report.md`, `paper_observation_report.md`, `paper_observation_policy_card.json`, and `stage_gate_report.md` before recommending QMT readonly.
If the repair branch has zero trades, do not recommend QMT or paper promotion; use `repair_dsl_next_actions.md` to relax the DSL and rerun the branch.
When `repair_dsl_auto_repair.md` exists, treat it as ranked research candidates only. Prefer candidates with `audit_status: VALID` and nonzero trades, then rerun the chosen candidate through paper observation and stage gates before QMT readonly.
If a ranked repair candidate has `audit_status: VALID`, `paper_status: VALID`, and enough trades, run `qmt-config-status` first.
